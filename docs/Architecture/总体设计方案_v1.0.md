## 1. 设计概述

### 1.1 系统定位

慕思IoT设备管理平台是面向慕思内部的物联网设备全生命周期管理系统，对标华为DP平台和涂鸦设备开发平台，核心解决三个问题：

1. **产品标准化** — 通过物模型规范统一设备数据格式，消除每款产品单独开发接入协议的现状
2. **快速接入** — 基于品类物模型模板机制，将新产品接入周期从周级缩短到天级
3. **统一运维** — 集中管理产品、设备、固件，提供实时监控和OTA升级能力

### 1.2 设计目标

| 维度 | 目标 | 量化指标 |
|------|------|----------|
| 性能 | API响应快、消息延迟低 | API P99 < 200ms，消息延迟 < 100ms |
| 容量 | 支撑慕思全线智能产品 | 10万+设备同时在线 |
| 可用性 | 生产级稳定性 | 年可用性 > 99.9% |
| 可扩展 | 支持新品类和新协议扩展 | 微服务独立扩缩容 |
| 安全 | 设备通信和数据安全 | 敏感数据100%加密 |

### 1.3 设计约束

- 技术栈锁定：Spring Boot 3.2 + Vue 3 + MySQL 8.0
- 部署环境：Docker + Kubernetes
- 协议支持：MVP阶段优先MQTT和BLE，后续扩展CoAP/HTTP
- 团队规模：小团队，优先开发效率，避免过度设计

### 2.2 架构分层职责

| 层级 | 职责 | 关键技术 |
|------|------|----------|
| 接入层 | 用户/设备接入，协议适配 | Nginx, EMQX(MQTT Broker) |
| 网关层 | 统一入口，路由转发，安全拦截 | Spring Cloud Gateway, JWT |
| 服务层 | 业务逻辑处理，领域模型 | Spring Boot 3.2, JPA |
| 中间件层 | 异步解耦，缓存加速，日志检索 | Kafka 3.6, Redis 7.2, ES 8.12 |
| 数据层 | 持久化存储 | MySQL 8.0, MinIO, InfluxDB 2.7 |
| 基础设施层 | 服务治理，可观测性，容器编排 | Nacos, SkyWalking, K8s |

### 2.3 服务间通信

```
                    同步调用 (REST/Feign)
product-service ◄──────────────────────► thing-model-service
       │                                        │
       │         异步消息 (Kafka)                │
       ▼                                        ▼
  ┌─────────┐    device.registered         ┌─────────┐
  │  Kafka   │◄────────────────────────────│ device-  │
  │  Topics  │    ota.task.created         │ service  │
  │          │────────────────────────────►│          │
  └─────────┘    firmware.published        └─────────┘
       │                                        ▲
       └────────────────────────────────────────┘
                    ota-service
```

**通信策略**：
- 查询类操作：同步REST调用（通过Feign Client）
- 状态变更通知：异步Kafka消息（最终一致性）
- 设备数据上报：Kafka → 各消费服务

**核心Kafka Topic**：

| Topic | 生产者 | 消费者 | 说明 |
|-------|--------|--------|------|
| `device.property.report` | 设备接入层 | device-service | 设备属性上报 |
| `device.event.report` | 设备接入层 | device-service | 设备事件上报 |
| `device.status.change` | device-service | 仪表盘/告警 | 设备上下线 |
| `ota.task.progress` | device-service | ota-service | OTA升级进度 |
| `product.published` | product-service | device-service | 产品发布通知 |

## 3. 微服务设计

### 3.1 服务划分

| 服务 | 端口 | 核心职责 | 数据库 | 依赖服务 |
|------|------|----------|--------|----------|
| product-service | 8082 | 产品CRUD、品类管理、产品发布 | product-db | thing-model-service |
| thing-model-service | 8083 | 物模型CRUD、模板管理、版本管理、功能点管理 | thing-model-db | — |
| device-service | 8084 | 设备注册/认证、上下线、属性上报、设备监控 | device-db | product-service |
| ota-service | 8085 | 固件管理、升级任务、进度监控 | ota-db | device-service |
| user-service | 8081 | 用户管理、角色权限、JWT签发 | user-db | — |

### 3.2 product-service 产品管理服务

**领域模型**：

```
Product (产品)
├── id, pid(PID_XXXXXX), name, model, brand
├── category(智能床垫/电动床/智能枕头)
├── protocol(MQTT/BLE)
├── status(DEVELOPING → PUBLISHED)
├── createTime, updateTime
│
ProductCategory (产品分类)
├── id, name, description, icon
└── thingModelTemplateId
```

**核心业务规则**：
- PID自动生成：`PID_` + 6位大写字母数字随机串，全局唯一
- 产品型号格式：`DR-{品类缩写}-{序号}`（如DR-M001）
- 状态流转：DEVELOPING → PUBLISHED（不可逆）
- 发布前置条件：至少配置1个物模型功能点
- 已发布产品：基本信息锁定不可编辑，物模型仍可编辑

**API端点**：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/products | 产品列表（分页/筛选） |
| POST | /api/products | 创建产品 |
| GET | /api/products/{id} | 产品详情 |
| PUT | /api/products/{id} | 更新产品（DEVELOPING状态） |
| DELETE | /api/products/{id} | 删除产品（DEVELOPING状态） |
| POST | /api/products/{id}/publish | 发布产品 |
| GET | /api/products/generate-pid | 生成唯一PID |
| GET/POST/PUT/DELETE | /api/categories | 品类CRUD |

### 3.3 thing-model-service 物模型服务

**领域模型**：

```
ThingModel (物模型)
├── id, productId, version(语义化版本), status(DRAFT/PUBLISHED)
├── propertiesJson, eventsJson, commandsJson
└── changeLog
│
ThingModelPoint (功能点)
├── id, thingModelId, pointId(标识符), name
├── pointType(PROPERTY/EVENT/COMMAND)
├── dataType(int/float/bool/string/enum)
├── access(readOnly/writeOnly/readWrite) — 仅PROPERTY
├── unit, range{min,max}, enumValues[{value,description}]
├── defaultValue, description
│
ThingModelTemplate (品类模板)
├── id, category, name, description
└── points[] — 预置功能点列表
```

**核心业务规则**：
- 功能点标识符：小写英文+数字+下划线，同物模型内唯一
- 数据类型决定可编辑字段：int/float(单位/范围/step)、bool(trueLabel/falseLabel)、string(maxLength)、enum(枚举值列表)
- 版本号：首次v1.0.0，每次发布次版本+1
- 品类模板导入：批量写入功能点到产品物模型
- 导出JSON：驼峰命名，文件名`{型号}_{版本}.json`

### 3.4 device-service 设备管理服务

**领域模型**：

```
Device (设备)
├── id, name, deviceKey(DK_PID前6位_8位HEX), serialNumber
├── productId, productName, thingModelId — 关联产品（自动继承）
├── deviceType, status(ONLINE/OFFLINE/DISABLED)
├── firmwareVersion, properties(JSON)
├── lastOnlineTime, createTime, updateTime
│
DeviceShadow (设备影子)
├── deviceId, properties(Desired), reportedProperties(Reported)
├── metadata, version
```

**核心业务规则**：
- 注册设备：选品类 → 选已发布产品 → 自动继承thingModelId
- DEVELOPING产品不可关联设备
- deviceKey自动生成：`DK_` + PID前6位 + `_` + 8位随机HEX
- 设备属性按关键词自动分组：传感器/控制/其他

### 3.5 ota-service OTA管理服务

**领域模型**：

```
Firmware (固件)
├── id, version, productId, productName
├── fileUrl, fileSize, status(DRAFT/PUBLISHED/DEPRECATED)
├── changeLog, createTime
│
OtaTask (升级任务)
├── id, name, firmwareId, targetVersion
├── totalDevices, successCount, failCount
├── status(PENDING/RUNNING/COMPLETED/CANCELLED)
├── description, createTime
```

**固件状态流转**：`DRAFT → PUBLISHED → DEPRECATED`

**任务状态流转**：`PENDING → RUNNING → COMPLETED/CANCELLED`

## 4. 数据架构设计

### 4.1 数据库拆分策略

每个微服务独立数据库，遵循Database-per-Service模式：

| 服务 | 数据库 | 核心表 | 存储引擎 |
|------|--------|--------|----------|
| product-service | product-db | product, product_category | MySQL/H2 |
| thing-model-service | thing-model-db | thing_model, thing_model_point, thing_model_template | MySQL/H2 |
| device-service | device-db | device, device_shadow, device_property, device_event, device_log | MySQL/H2 |
| ota-service | ota-db | firmware, ota_task, ota_task_device | MySQL/H2 |
| user-service | user-db | sys_user, sys_role, sys_user_role, sys_operate_log | MySQL/H2 |

> MVP阶段使用H2文件模式（`./data/{service}-db`），生产环境切换MySQL集群。

### 4.2 核心ER关系

```
product 1──N device          产品下有多个设备
product 1──1 thing_model     产品关联一个物模型
thing_model 1──N thing_model_point   物模型包含多个功能点
device 1──1 device_shadow    设备对应一个影子
firmware N──1 product        固件归属产品
ota_task N──1 firmware       升级任务关联固件
ota_task 1──N ota_task_device  任务包含多个设备明细
```

### 4.3 数据量估算与分表策略

| 表 | 预估数据量 | 策略 |
|----|-----------|------|
| product | 100~1,000 | 单表 |
| thing_model / point | 1,000~10,000 | 单表 |
| device | 10万~100万 | 考虑分库 |
| device_property | 1,000万+/月 | 按device_id哈希分16~32表 |
| device_event | 100万+/月 | 按device_id哈希分表 |
| device_log | 500万+/月 | 按月分表，保留6个月 |

### 4.4 缓存设计

| 缓存Key | 数据 | TTL | 更新策略 |
|---------|------|-----|----------|
| `product:{id}` | 产品详情 | 30min | 更新时删除 |
| `thing-model:{productId}` | 物模型JSON | 30min | 发布时删除 |
| `device:shadow:{deviceId}` | 设备影子 | 实时 | 上报时更新 |
| `device:online:{deviceId}` | 在线状态 | 5min | 心跳续期 |
| `ota:task:{taskId}:progress` | 任务进度 | 实时 | 进度更新时写入 |

### 4.5 时序数据存储

设备属性时序数据写入InfluxDB：

```
measurement: device_property
tags: device_id, product_id, identifier
fields: value (根据数据类型)
time: 上报时间戳

保留策略:
- 原始数据: 30天
- 1分钟聚合: 90天
- 1小时聚合: 1年
```

## 5. 设备接入架构

### 5.1 协议接入层

```
                    设备终端
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │MQTT Broker│ │BLE Gateway│ │HTTP Server│
    │ (EMQX)   │ │           │ │           │
    │ :1883/8883│ │           │ │ :8443     │
    └─────┬────┘ └─────┬────┘ └─────┬────┘
          │            │            │
          └────────────┼────────────┘
                       ▼
              ┌────────────────┐
              │ 协议适配层      │
              │ Protocol       │
              │ Adapter        │
              │ (统一消息格式)  │
              └───────┬────────┘
                      ▼
              ┌────────────────┐
              │ Kafka          │
              │ device.*.report│
              └───────┬────────┘
                      ▼
              ┌────────────────┐
              │ device-service │
              │ (消息消费处理)  │
              └────────────────┘
```

### 5.2 设备认证流程

```
设备 → Auth(ProductKey, DeviceKey, DeviceSecret) → 平台
平台 → 验证三元组 → 返回MQTT连接信息
设备 → MQTT Connect(ClientID=DK, Username=PK, Password=DS) → EMQX
EMQX → 连接确认 → 设备上线
```

### 5.3 MQTT Topic规范

| Topic | 方向 | 说明 |
|-------|------|------|
| `/{productKey}/{deviceKey}/property/report` | 设备→云 | 属性上报 |
| `/{productKey}/{deviceKey}/event/report` | 设备→云 | 事件上报 |
| `/{productKey}/{deviceKey}/command/invoke` | 云→设备 | 命令下发 |
| `/{productKey}/{deviceKey}/command/reply` | 设备→云 | 命令响应 |
| `/{productKey}/{deviceKey}/ota/notify` | 云→设备 | OTA升级通知 |
| `/{productKey}/{deviceKey}/ota/progress` | 设备→云 | OTA进度上报 |

### 5.4 消息格式

```json
{
  "id": "msg_001",
  "version": "1.0",
  "method": "thing.property.report",
  "params": {
    "temperature": { "value": 25, "time": 1705312200000 },
    "humidity": { "value": 60, "time": 1705312200000 }
  }
}
```

## 7. 安全架构设计

### 7.1 安全分层

| 层级 | 措施 |
|------|------|
| 网络层 | VPC隔离、安全组、DDoS防护、WAF |
| 传输层 | TLS 1.3(Web/API)、TLS 1.2(MQTT)、全链路HTTPS |
| 应用层 | JWT认证(RS256)、RBAC权限、API限流(100QPS/IP) |
| 数据层 | 密码BCrypt、设备密钥AES-256、敏感字段加密、审计日志 |

### 7.2 认证体系

**用户认证**：
- JWT Token（RS256签名），Access Token 1小时，Refresh Token 7天
- 密码策略：8位+大小写+数字+特殊字符，90天强制更换，5次错误锁定30分钟

**设备认证**：
- 三元组认证：ProductKey + DeviceKey + DeviceSecret
- 支持密钥认证（通用）和证书认证（高安全场景）

### 7.3 权限模型 (RBAC)

| 角色 | 权限范围 |
|------|----------|
| 系统管理员 | 全部权限 |
| 产品经理 | 产品/物模型 创建/编辑/发布 |
| 研发工程师 | 物模型/设备 只读 + 命令下发 |
| 运维人员 | 设备/OTA 管理 |
| 测试工程师 | 全部只读 |

## 9. 核心业务流程

### 9.1 产品创建到发布

```
创建产品(填写基本信息+选品类+选协议)
    → 系统生成PID，状态=DEVELOPING
    → 进入产品详情页
    → 从品类模板导入物模型功能点
    → 编辑/添加/删除功能点
    → 预览/导出物模型JSON
    → 点击发布 → 检查至少1个功能点
    → 状态=PUBLISHED，基本信息锁定
```

### 9.2 设备注册到监控

```
注册设备(选品类 → 选已发布产品 → 填设备名/SN)
    → 自动继承productId/productName/thingModelId
    → 自动生成deviceKey
    → 设备三元组认证 → MQTT连接
    → 属性上报 → Kafka → device-service → 更新影子/写入时序库
    → 设备监控页：实时属性 + 告警Timeline + 命令下发
```

### 9.3 OTA升级流程

```
上传固件(选产品+版本号+文件) → 状态=DRAFT
    → 发布固件 → 状态=PUBLISHED
    → 创建升级任务(选固件+填任务名)
    → 启动任务 → 下发OTA通知到目标设备
    → 设备下载固件 → 安装 → 上报进度
    → 任务监控：进度条 + 成功/失败统计
```

## 12. 关键设计决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 微服务拆分粒度 | 5个核心服务 | 按业务域拆分，小团队可维护 |
| 服务间通信 | REST(同步) + Kafka(异步) | 查询用REST，事件用Kafka解耦 |
| 数据库策略 | Database-per-Service | 服务独立演进，避免耦合 |
| 物模型存储 | 功能点独立表 + JSON字段 | 结构化查询 + 灵活扩展 |
| 设备接入协议 | MQTT优先 | IoT行业标准，EMQX生态成熟 |
| 前端UI框架 | Element Plus | Vue3原生适配，中文文档完善 |
| MVP数据库 | H2文件模式 | 零运维成本，快速迭代 |
| 缓存策略 | Cache-Aside | 简单可靠，适合读多写少场景 |
