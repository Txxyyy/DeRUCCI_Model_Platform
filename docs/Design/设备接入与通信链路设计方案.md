# 设备接入与通信链路设计方案

> 版本: v1.0 | 日期: 2026-03-23 | 状态: 待评审

## 1. 概述

### 1.1 背景

慕思IoT平台MVP已完成产品管理、物模型管理、设备基础CRUD和OTA管理。但设备认证、MQTT通信、产线对接等核心链路尚未实现。本文档定义设备从生产到通信的全链路方案。

### 1.2 适用产品

智能床垫、智能枕头、电动床等慕思智能家居产品。特点：
- 出货量中等（年产万级～十万级）
- 产品生命周期长（5-10年）
- 单价高，安全要求高

### 1.3 关键设计决策

| 编号 | 决策 | 理由 |
|------|------|------|
| D1 | 一机一密（非一型一密） | 单价高、量不大，一机一密管理成本可接受；一型一密固件被逆向则全线沦陷 |
| D2 | PID对外展示，ProductKey仅用于认证 | PID短且可读，适合蓝牙广播和APP展示；PK长且随机，适合安全场景 |
| D3 | 设备不拉取物模型 | 固件编译时确定能力，物模型供平台校验和APP渲染 |
| D4 | 认证与MQTT连接分离 | 先HTTPS认证拿Token，再用Token连EMQX；Broker不直接查库 |
| D5 | MQTT Broker选用EMQX | 国产开源，企业版功能丰富，支持认证插件和规则引擎 |

## 2. 标识体系

### 2.1 四层标识设计

| 标识 | 层级 | 格式 | 用途 | 保密性 |
|------|------|------|------|--------|
| PID | 产品级（同型号共享） | `PID_XXXXXX` (10位) | 对外产品标识：蓝牙广播、APP识别、MQTT Topic | 公开 |
| ProductKey | 产品级（同型号共享） | `PK_` + 18位随机 (20位) | 设备认证签名计算 | 保密 |
| DeviceKey | 设备级（每台唯一） | `DK_` + 18位随机 (20位) | 设备唯一标识、MQTT ClientID | 半公开 |
| DeviceSecret | 设备级（每台唯一） | 32位随机字符串 | 设备认证签名计算 | 保密(L4) |

### 2.2 辅助标识

| 标识 | 说明 |
|------|------|
| SN码 | 外壳印刷的序列号，用户扫码绑定设备 |
| MAC地址 | 模组硬件标识，配网时APP发现设备 |
| category | 产品品类（智能床垫/智能枕头/电动床），APP加载对应UI模板 |
| deviceType | TEST / PRODUCTION，区分调试设备和正式设备 |

### 2.3 标识归属关系

```
Product (产品)
  ├── pid = "PID_A1B2C3"              ← 对外标识
  ├── productKey = "PK_xR7kM9pQ2wE5"  ← 认证密钥
  ├── name = "慕思智能床垫Pro"
  ├── category = "智能床垫"
  ├── blePrefix = "DeRUCCI_A1B2C3"     ← 蓝牙广播前缀
  │
  └── Device (设备) × N
        ├── deviceKey = "DK_m3Nq8Rw2Xp"   ← 设备标识
        ├── deviceSecret = "a7B9c2D4..."    ← 设备密钥(AES-256加密存储)
        ├── serialNumber = "SN2026032300001"
        └── mac = "AA:BB:CC:DD:EE:FF"
```

## 3. 产线烧录流程

### 3.1 方案：一机一密 + 产线动态烧录

模组在烧录前是"白板"，所有凭证由烧录工具作为中间人从平台获取后写入。

### 3.2 模组 Flash 分区布局

| 分区 | 内容 | 说明 |
|------|------|------|
| Bootloader区 | 安全启动引导 | 固定 |
| 固件区(OTA-A) | 应用固件（业务逻辑） | OTA主分区 |
| 固件区(OTA-B) | OTA备份分区 | OTA回滚用 |
| 安全存储区(eFuse/加密Flash) | PID、ProductKey、DeviceKey、DeviceSecret、SN | 硬件级防读取 |
| 数据区 | 用户配置、运行时数据 | 可擦写 |

### 3.3 固件内置内容（编译时写死，所有同型号共用）

| 内容 | 说明 |
|------|------|
| 平台认证地址 | `https://auth.derucci.com/api/auth/device` |
| MQTT默认端口 | 8883 (TLS) |
| CA根证书 | 验证平台TLS证书合法性 |
| 固件版本号 | OTA版本比对 |
| 蓝牙广播名模板 | `DeRUCCI_{PID去前缀}_{SN后4位}` |
| 通信协议实现 | MQTT客户端、认证握手、重连策略、心跳 |
| 配网模块 | WiFi配网（SmartConfig/BLE配网/AP配网） |

### 3.4 产线动态写入内容（每台设备不同）

| 内容 | 来源 | 写入方式 |
|------|------|---------|
| PID | 产品创建时生成 | 烧录工具从平台获取 |
| ProductKey | 产品创建时生成 | 烧录工具从平台获取 |
| DeviceKey | 设备注册时平台生成 | 烧录工具写入 |
| DeviceSecret | 设备注册时平台生成 | 烧录工具写入（明文仅此一次） |
| SN码 | 产线系统分配 | 烧录工具写入 |

### 3.5 产线烧录时序

```
产线工位         烧录工具(PC软件)              慕思IoT平台
   │                  │                           │
   │ ① 扫描模组条码    │                           │
   │ (获取MAC地址)     │                           │
   │ ────────────────>│                           │
   │                  │                           │
   │ ② 扫描外壳SN码   │                           │
   │ ────────────────>│                           │
   │                  │                           │
   │                  │ ③ 调用平台API注册设备        │
   │                  │ POST /api/devices          │
   │                  │ {productId, name,          │
   │                  │  serialNumber, mac,        │
   │                  │  deviceType: "PRODUCTION"} │
   │                  │ ──────────────────────────>│
   │                  │                           │
   │                  │                           │ ④ 平台自动生成:
   │                  │                           │   deviceKey = "DK_..."
   │                  │                           │   deviceSecret = "..."(32位)
   │                  │                           │   状态 = UNACTIVE
   │                  │                           │
   │                  │ ⑤ 返回完整凭证              │
   │                  │ {deviceKey, deviceSecret,  │
   │                  │  productKey}               │
   │                  │ <──────────────────────────│
   │                  │                           │
   │ ⑥ 串口写入模组    │                           │
   │ 安全存储区:       │                           │
   │  PID+PK+DK+DS+SN│                           │
   │ <────────────────│                           │
   │                  │                           │
   │ ⑦ 模组自检通过    │                           │
   │ ────────────────>│ ⑧ 标记烧录完成              │
   │                  │ ──────────────────────────>│
```

### 3.6 批量预注册（提高产线效率）

```
烧录工具                              平台
   │                                   │
   │ POST /api/devices/batch-register  │
   │ {productId, count: 100,           │
   │  snPrefix: "SN2026032300"}        │
   │ ─────────────────────────────────>│
   │                                   │ 生成100组三元组
   │ 返回100组凭证                      │
   │ [{sn, pk, dk, ds}, ...]           │
   │ <─────────────────────────────────│
   │                                   │
   │ 缓存在本地，逐台烧录               │
```

### 3.7 安全要点

- DeviceSecret 只在注册时明文返回一次，平台只存AES-256加密值
- 模组安全存储区（eFuse/加密Flash）硬件级防读取
- 烧录工具需平台管理员Token授权，不留存密钥
- 烧录工具与平台之间走 HTTPS + IP白名单（仅允许工厂网络）

## 4. 设备认证流程

### 4.1 认证时序（设备首次激活）

```
设备(模组)                         平台认证服务              EMQX Broker
   │                                  │                       │
   │ ① 用户配网，设备联网              │                       │
   │                                  │                       │
   │ ② 从安全存储区读取凭证            │                       │
   │    PID, PK, DK, DS               │                       │
   │                                  │                       │
   │ ③ HTTPS POST /api/auth/device    │                       │
   │ {                                │                       │
   │   "pid": "PID_A1B2C3",          │                       │
   │   "deviceKey": "DK_m3Nq...",    │                       │
   │   "timestamp": 1711152000,       │                       │
   │   "nonce": "a1b2c3",            │                       │
   │   "sign": SHA256(PK+DK+DS       │                       │
   │           +timestamp+nonce),     │                       │
   │   "signMethod": "SHA256"         │                       │
   │ }                                │                       │
   │ ────────────────────────────────>│                       │
   │                                  │                       │
   │                                  │ ④ 验证流程:            │
   │                                  │  查PID→找产品→确认已发布│
   │                                  │  查DK→找设备→确认未禁用 │
   │                                  │  取PK+解密DS→重算sign  │
   │                                  │  比对sign→验证通过      │
   │                                  │  检查timestamp±5min    │
   │                                  │  生成MQTT Token(24h)   │
   │                                  │                       │
   │ ⑤ 返回MQTT连接信息               │                       │
   │ {                                │                       │
   │   "mqttBroker": "mqtt.derucci.com",                      │
   │   "mqttPort": 8883,              │                       │
   │   "mqttClientId": "DK_m3Nq...", │                       │
   │   "mqttUsername": "PID_A1B2C3",  │                       │
   │   "mqttPassword": "<jwt_token>", │                       │
   │   "tokenExpireAt": 1711238400,   │                       │
   │   "topics": {                    │                       │
   │     "propReport": "/{PID}/{DK}/property/report",         │
   │     "eventReport": "/{PID}/{DK}/event/report",           │
   │     "cmdInvoke": "/{PID}/{DK}/command/invoke",           │
   │     "cmdReply": "/{PID}/{DK}/command/reply",             │
   │     "otaNotify": "/{PID}/{DK}/ota/notify",               │
   │     "otaProgress": "/{PID}/{DK}/ota/progress"            │
   │   }                              │                       │
   │ }                                │                       │
   │ <────────────────────────────────│                       │
   │                                  │                       │
   │ ⑥ TLS连接MQTT Broker             │                       │
   │ ─────────────────────────────────────────────────────────>│
   │                                  │                       │
   │ ⑦ 订阅命令Topic                  │                       │
   │ ─────────────────────────────────────────────────────────>│
   │                                  │                       │
   │ ⑧ 上报首次属性                    │                       │
   │ ─────────────────────────────────────────────────────────>│
   │                                  │                       │
   │                                  │ ⑨ 设备状态 UNACTIVE→ONLINE
```

注意：**DeviceSecret不直接传输**，而是用来计算签名。即使HTTPS被截获也拿不到密钥。

### 4.2 认证失败处理

| 错误码 | 场景 | 设备行为 |
|--------|------|---------|
| 401001 | 签名验证失败（密钥错误） | 停止重试，LED闪烁报错 |
| 401002 | 设备未注册 | 停止重试，LED闪烁报错 |
| 401003 | 设备已禁用 | 停止重试，LED闪烁报错 |
| 401004 | 产品未发布 | 停止重试，LED闪烁报错 |
| 401005 | timestamp超时（±5min） | 校准时间后重试 |
| 429001 | 请求频率过高 | 退避后重试 |
| 500001 | 服务端错误 | 指数退避重试 |

## 5. MQTT通信链路

### 5.1 EMQX Broker 部署

Docker Compose 新增 EMQX 服务，使用 HTTP Auth 插件对接平台认证服务。

### 5.2 Topic 规范

| Topic | 方向 | QoS | 说明 |
|-------|------|-----|------|
| `/{PID}/{DK}/property/report` | 设备→平台 | 1 | 属性上报 |
| `/{PID}/{DK}/event/report` | 设备→平台 | 1 | 事件上报 |
| `/{PID}/{DK}/command/invoke` | 平台→设备 | 1 | 命令下发 |
| `/{PID}/{DK}/command/reply` | 设备→平台 | 1 | 命令响应 |
| `/{PID}/{DK}/ota/notify` | 平台→设备 | 1 | OTA升级通知 |
| `/{PID}/{DK}/ota/progress` | 设备→平台 | 1 | OTA进度上报 |

### 5.3 MQTT 连接参数

| 参数 | 值 | 说明 |
|------|------|------|
| ClientID | DeviceKey | 设备唯一标识 |
| Username | PID | 产品标识 |
| Password | JWT Token | 认证接口返回的临时Token |
| Keep Alive | 60s | 心跳间隔 |
| Clean Session | true | 首次连接；重连时可用false恢复订阅 |

### 5.4 消息格式规范

所有上行/下行消息统一JSON格式：

```json
// 属性上报（设备→平台）
{
  "id": "msg_1711152000_001",
  "version": "1.0",
  "method": "thing.property.report",
  "params": {
    "body_pressure": { "value": [12,15,8,20], "time": 1711152000000 },
    "heart_rate": { "value": 72, "time": 1711152000000 },
    "sleep_stage": { "value": "deep", "time": 1711152000000 }
  }
}

// 命令下发（平台→设备）
{
  "id": "cmd_001",
  "version": "1.0",
  "method": "thing.command.invoke",
  "params": {
    "commandId": "adjust_firmness",
    "input": { "zone": "lumbar", "level": 3 }
  }
}

// 命令响应（设备→平台）
{
  "id": "cmd_001",
  "version": "1.0",
  "method": "thing.command.reply",
  "code": 0,
  "message": "success",
  "data": { "currentLevel": 3 }
}
```

### 5.5 心跳与重连策略

- **心跳**: MQTT Keep Alive = 60s，Broker检测不到心跳自动判定离线
- **断线重连**: 指数退避 `min(1s × 2^attempt + random(0,1000ms), 60s)`
- **重连场景**:
  - 网络断开 → 直接重连MQTT（Token未过期）
  - MQTT被踢 → 重新认证 → 重连MQTT
  - Token过期 → 重新认证 → 重连MQTT
  - 认证失败(密钥错误) → 停止重连，上报错误
  - 认证失败(设备被禁用) → 停止重连，上报错误

## 6. APP配网与绑定

### 6.1 产品配置缓存

APP启动时拉取已发布产品列表，缓存到本地：

```
GET /api/products/published
→ [{pid, name, category, model, imageUrl, blePrefix}, ...]
```

支持 `If-Modified-Since` 头，减少无效请求。

### 6.2 蓝牙广播设计

```
蓝牙广播名格式: DeRUCCI_{PID去前缀}_{SN后4位}
示例: DeRUCCI_A1B2C3_0001 (19字符，蓝牙名上限29字符)

蓝牙广播数据（Manufacturer Specific Data）:
  ├── Company ID: 自定义
  ├── PID: "PID_A1B2C3"
  ├── SN: "SN2026032300001"
  ├── FW Version: "1.0.0"
  └── Config Status: 0x00=未配网 / 0x01=已配网
```

### 6.3 APP配网绑定时序

```
用户手机APP                     设备(床垫)                    平台
   │                              │                           │
   │ ① 打开APP，点"添加设备"       │                           │
   │    选择品类"智能床垫"          │                           │
   │                              │                           │
   │ ② BLE扫描附近设备             │                           │
   │    过滤前缀 "DeRUCCI_"        │                           │
```

### 6.4 APP设备类型区分

APP通过 `category` 字段加载对应UI模板：
- `智能床垫` → 床垫控制面板（硬度调节、睡眠报告、温度）
- `智能枕头` → 枕头控制面板（高度调节、打鼾干预）
- `电动床` → 电动床面板（角度调节、零重力模式）

物模型定义具体控件：属性→数字/滑块/图表，命令→按钮/开关。

### 7.3 核心API（C语言，适配嵌入式）

```c
/* ========== 初始化与连接 ========== */

typedef struct {
```

### 7.4 物模型与固件的关系

设备固件不拉取物模型。固件在编译时就确定了自身能力：

```
固件代码（编译时写死）：
  - 温度传感器 → 每30秒读取 → 上报 "temperature": 25.5
  - 压力传感器 → 每10秒读取 → 上报 "body_pressure": [12,15,8,...]
  - 硬度调节   → 收到 "adjust_firmness" 命令 → 驱动电机

物模型（平台侧）：
  - 平台校验上报数据是否合法（类型、范围）
  - APP根据物模型渲染对应控件（温度→数字、压力→热力图、硬度→滑块）

关键：两边的标识符必须一致（固件上报 "temperature" ↔ 物模型定义 "temperature"）
```

## 8. 设备生命周期

### 8.1 状态机

```
                 ┌──────────┐
 产线注册 ──────>│ UNACTIVE │ (已有三元组，从未联网)
                 └────┬─────┘
                      │ 首次认证成功
                      v
                 ┌──────────┐
           ┌────>│  ONLINE  │<────┐
           │     └────┬─────┘     │
           │          │ 心跳超时/  │ 重新连接
           │          │ 主动断开   │
           │     ┌────v─────┐     │
           │     │  OFFLINE │─────┘
           │     └────┬─────┘
           │          │ 管理员禁用
           │     ┌────v─────┐
           └─────│ DISABLED │ (密钥失效，拒绝认证)
                 └────┬─────┘
                      │ 注销
                 ┌────v─────┐
                 │ DELETED  │ (软删除，保留审计记录)
                 └──────────┘
```

### 8.2 状态转换触发条件

| 转换 | 触发条件 |
|------|---------|
| UNACTIVE → ONLINE | 设备首次认证成功 |
| ONLINE → OFFLINE | 心跳超时 / 设备主动断开 / 网络异常 |
| OFFLINE → ONLINE | 设备重新连接并认证 |
| ONLINE/OFFLINE → DISABLED | 管理员手动禁用 / 密钥泄露应急 |
| DISABLED → ONLINE | 管理员解除禁用 + 设备重新认证 |
| 任意状态 → DELETED | 管理员注销设备（软删除） |

## 9. 安全设计

### 9.1 安全分层

| 层级 | 措施 | 优先级 |
|------|------|--------|
| 传输层 | TLS 1.2+ 加密MQTT连接 (端口8883) | MVP |
| 认证层 | 三元组签名验证 + 临时JWT Token | MVP |
| 消息层 | timestamp + nonce 防重放（±5min窗口） | MVP |
| 存储层 | DeviceSecret AES-256加密存储 | MVP |
| 应急 | 单设备密钥重置 + 批量禁用 | MVP |
| 增强 | X.509证书双向认证 | 二期 |
| 增强 | 安全启动(Secure Boot) | 三期 |

### 9.2 密钥泄露应急流程

1. 管理员在平台禁用该设备 → 状态变为 DISABLED
2. EMQX踢掉该设备的MQTT连接
3. 重置DeviceSecret → `POST /api/devices/{id}/reset-secret`
4. 产线重新烧录新密钥（或远程OTA更新密钥，如果支持）
5. 解除禁用，设备用新密钥重新认证

- 密钥自动轮换

- Secure Boot支持
