# 慕思通用物模型管理平台 PRD v2.0

## 1. 产品概述

### 1.1 产品背景

随着慕思智能化战略的推进，越来越多的智能产品（智能床垫、智能枕头、智能睡眠监测设备等）需要接入统一的管理平台。当前的设备接入缺乏标准化管理，导致：
- 每款产品都需要单独开发接入协议
- 设备数据格式不统一，难以统一分析
- 新产品接入周期长，成本高

因此，需要建设一个**通用物模型管理平台**，实现产品的标准化管理和快速接入。

### 1.2 产品定位

慕思内部使用的物联网设备管理平台，类比华为DP平台、涂鸦设备开发平台，专注于：
- 产品信息的统一管理
- 物模型的标准化定义
- 设备的注册接入与实时监控
- 固件OTA升级管理
- 运营数据可视化仪表盘

### 1.3 目标用户

| 用户角色 | 使用场景 |
|---------|---------|
| 产品经理 | 创建产品、定义物模型、管理品类模板、发布产品 |
| 研发工程师 | 查看物模型规范、注册设备、导出物模型JSON、开发固件/APP |
| 运维人员 | 监控设备状态、管理OTA升级、查看仪表盘 |
| 测试工程师 | 验证物模型功能、测试设备接入、设备调试 |

### 1.4 核心价值

1. **标准化**：统一的物模型规范，设备数据格式标准化
2. **快速接入**：基于品类物模型快速创建新产品的物模型，缩短接入周期
3. **统一管理**：集中管理所有产品、设备和固件版本
4. **可扩展**：支持多种协议和设备类型

## 3. 功能需求

### 3.1 产品管理

#### 3.1.1 产品创建

**功能描述**：用户创建新产品，系统自动分配唯一PID，产品创建后初始状态为"开发中"

**表单字段**：
| 字段 | 类型 | 必填 | 验证规则 | 说明 |
|-----|------|------|---------|------|
| 产品型号 | 字符串 | 是 | 1-50字符，英文字母数字连字符，不能重复 | 如 "DR-M001" |
| 产品名称 | 字符串 | 是 | 1-50字符，不能重复 | 如 "慕思智能床垫X1" |
| 产品品牌 | 字符串 | 是 | 1-50字符 | 如 "慕思" |
| 通信方式 | 下拉选择 | 是 | 必选：MQTT/BLE | 设备接入协议 |
| 产品品类 | 下拉选择 | 是 | 必选：智能床垫/电动床/智能枕头 | 产品大类，决定品类物模型 |

**产品品类选项**：
| 品类值 | 说明 | 品类物模型 |
|-------|------|----------|
| 智能床垫 | 慕思智能床垫产品 | 智能床垫标准物模型 |
| 电动床 | 慕思电动床产品 | 电动床标准物模型 |
| 智能枕头 | 慕思智能枕头产品 | 智能枕头标准物模型 |

**说明**：产品经理在「品类物模型」页面维护各品类的标准物模型。创建产品时选择品类，产品自动携带该品类的物模型，可在产品详情中选择添加需要的物模型点。

**通信方式选项**：
| 协议 | 说明 | 适用场景 |
|-----|------|---------|
| MQTT | 长连接协议，适合实时性要求高的设备 | 智能床垫、电动床 |
| BLE | 蓝牙低功耗，适合短距离设备 | 智能枕头、手环等 |

**PID生成规则**：
- 格式：`PID_` + 6位大写字母数字随机字符串
- 示例：`PID_A1B2C3`、`PID_X9Y8Z7`
- 生成方式：系统自动生成，确保全局唯一
- 用途：用于设备接入时的ProductKey标识

**产品型号格式规范**：
- 格式：`品牌缩写-品类缩写-序号`
- 品牌缩写：DR（DeRUCCI慕思）
- 品类缩写：M（Mattress床垫）、B（Bed床）、P（Pillow枕头）
- 序号：3位数字，如001、002
- 示例：DR-M001、DR-B001、DR-P001

**用户交互流程**：
```
1. 用户点击"新建产品"按钮
2. 系统显示产品创建表单
3. 用户填写产品基本信息（型号、名称，品牌）
   - 产品型号支持输入联想提示（如输入"DR-"显示格式提示）
4. 用户选择通信方式
5. 用户选择产品品类
6. 用户点击"创建"按钮
7. 系统验证表单数据
   - 验证通过：创建产品，生成唯一PID，状态为"开发中"，返回成功提示
   - 验证失败：显示错误提示，停留在当前页面
8. 创建成功后，跳转到产品列表页，显示新创建的产品卡片
```

**字段详细验证规则**：
| 字段 | 验证规则 | 错误提示 |
|-----|---------|---------|
| 产品型号 | 只能包含英文字母、数字、连字符，不能以连字符开头或结尾 | "产品型号格式不正确，只能包含字母、数字和连字符" |
| 产品型号唯一性 | 数据库查询确认不重复 | "产品型号已存在，请使用其他型号" |
| 产品名称唯一性 | 数据库查询确认不重复 | "产品名称已存在，请使用其他名称" |

#### 3.1.3 产品状态管理

**产品状态说明**：
| 状态 | 英文 | 说明 | 产品信息可编辑 | 物模型可编辑 |
|-----|------|------|-------------|------------|
| 开发中 | DEVELOPING | 产品刚创建时的状态 | 是 | 是 |
| 已发布 | PUBLISHED | 产品经理点击发布按钮后 | 否 | 是 |

**状态流转规则**：
```
开发中 → 发布按钮 → 已发布（不可逆）
```

**发布流程**：
```
1. 产品经理在产品详情页点击"发布"按钮
2. 系统检查物模型是否已配置（至少有一个功能点）
   - 如果没有：提示"请先配置物模型后再发布"
3. 确认发布后，产品状态变更为"已发布"
4. 已发布状态的产品信息（型号、名称、品牌、通信方式、品类）锁定，不可编辑
5. 物模型仍然可以编辑和发布新版本
```

#### 3.1.4 产品详情页

**功能描述**：展示产品完整信息，管理物模型

**功能详情**：

1. **产品基本信息**
   - 显示所有产品信息
   - **编辑按钮**：仅在"开发中"状态时显示，可修改产品基本信息
   - **发布按钮**：仅在"开发中"状态时显示，点击后发布产品
   - **删除按钮**：仅在"开发中"状态时显示
   - 已发布状态：显示锁定图标，信息不可编辑

2. **物模型管理**
   - **品类物模型**：显示产品所属品类的标准物模型列表
   - **添加物模型**：从品类物模型中选择需要添加的功能点
   - **编辑功能点**：修改已有功能点的配置（根据数据类型显示不同字段）
   - **删除功能点**：移除不需要的功能点
   - **导出JSON**：导出物模型JSON用于APP和固件开发
   - **发布版本**：发布物模型版本

### 3.2 物模型管理

#### 3.2.1 物模型功能点

**功能描述**：每个物模型功能点都有唯一标识符，用于定义设备能力。支持根据数据类型动态编辑不同字段。

**物模型功能点结构**：

**通用字段**（所有类型都包含）：
| 字段 | 类型 | 必填 | 说明 |
|-----|------|------|------|
| 标识符(identifier) | 字符串 | 是 | 唯一标识，如 "temperature"，不能重复 |
| 功能名称(name) | 字符串 | 是 | 展示名称，如 "当前温度" |
| 功能类型(pointType) | 枚举 | 是 | 属性(PROPERTY)/事件(EVENT)/命令(COMMAND) |
| 数据类型(dataType) | 下拉选择 | 是 | int/float/bool/string/enum |
| 描述(description) | 字符串 | 否 | 功能说明 |

---

| 单位(unit) | 字符串 | 否 | 如 "℃"、"%"、"bpm" |
| 取值范围(range) | 对象 | 条件必填 | 最小值(min)和最大值(max) |
| 默认值(defaultValue) | 数值 | 否 | 符合取值范围 |

**enum 类型可编辑字段**：
| 字段 | 类型 | 必填 | 说明 |

**物模型ID命名规范**：
- 格式：小写英文字母、数字、下划线
- 规则：不能以数字开头，不能与同物模型内其他ID重复
- 示例：`current_temperature`、`target_hardness`、`sleep_mode`
- 建议：使用有意义的英文单词或缩写，采用下划线分隔

**数据类型默认值处理**：
| 数据类型 | 默认值 |
|---------|-------|
| int | 0 |
| float | 0.0 |
| bool | false |
| string | ""（空字符串） |
| enum | 第一个枚举值 |

**数据类型说明**：
| 类型 | 说明 | 适用场景 |
|-----|------|---------|
| int | 整数 | 温度、湿度、计数等 |
| float | 浮点数 | 精确温度、百分比等 |
| bool | 布尔值 | 开关状态、告警标志 |
| string | 字符串 | 文本信息、状态描述 |
| enum | 枚举 | 模式选择、档位设置 |

#### 3.2.2 品类物模型

**功能描述**：品类物模型由**产品经理**在「品类物模型」页面创建和维护，每个品类有标准物模型供产品使用。

**管理角色**：产品经理

**品类与物模型映射**：
| 产品品类 | 说明 |
|---------|------|
| 智能床垫 | 产品经理维护智能床垫品类的标准物模型 |
| 电动床 | 产品经理维护电动床品类的标准物模型 |
| 智能枕头 | 产品经理维护智能枕头品类的标准物模型 |

**品类物模型创建流程（产品经理角色）**：
```
1. 产品经理进入「品类物模型」页面
2. 选择要管理的品类（智能床垫/电动床/智能枕头）
3. 添加属性功能点（标识符、名称、数据类型、读写类型、单位、取值范围等）
4. 添加事件功能点
5. 添加命令功能点
6. 保存品类物模型
```

**产品使用品类物模型流程**：
```
1. 用户创建产品时选择品类
2. 系统自动将该品类的物模型添加到产品中
3. 用户进入产品详情页的物模型管理区域
4. 查看品类物模型列表
5. 选择需要启用的物模型点（勾选）
6. 对启用的物模型点进行编辑（根据数据类型编辑不同字段）
7. 可以新增额外的物模型点
8. 导出物模型JSON或发布版本
```

#### 3.2.3 物模型版本管理

**功能描述**：支持物模型多版本管理

**版本规则**：
| 规则 | 说明 |
|-----|------|
| 版本号格式 | 语义化版本 v主版本.次版本.修订号 |
| 初始版本 | 新建物模型默认为 v1.0.0 |
| 版本发布 | 每次发布自动递增次版本号 |
| 版本变更 | 记录变更内容，生成变更说明 |
| 版本回滚 | 可回滚到任意历史版本 |

**版本号递增规则**：
- 首次发布：v1.0.0
- 后续每次发布：次版本号+1，如 v1.1.0、v1.2.0
- 大版本更新（如有重大变更）：主版本号+1，如 v2.0.0

### 3.3 设备管理

#### 3.3.1 设备注册

**功能描述**：通过「选品类 → 选产品」二级联动将设备关联到已发布产品，自动继承物模型。

**注册表单字段**：
| 字段 | 类型 | 必填 | 说明 |
|-----|------|------|------|
| 品类选择 | RadioGroup | 是 | 智能床垫/电动床/智能枕头，联动加载产品列表 |
| 关联产品 | 卡片选择 | 是 | 展示该品类下已发布产品，DEVELOPING状态置灰不可选 |
| 设备名称 | 字符串 | 是 | 设备显示名称 |
| SN码(serialNumber) | 字符串 | 否 | 设备序列号 |

**自动填充字段**（选中产品后）：

- 只有 `PUBLISHED` 状态的产品可关联设备

```java
@Entity
@Table(name = "t_product")
public class Product {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, unique = true, length = 50)
    private String pid;  // 系统生成，唯一标识，如 "PID_ABC123"

    @Column(nullable = false, length = 50)
    private String name;  // 产品名称

    @Column(nullable = false, unique = true, length = 50)
    private String model;  // 产品型号，如 "DR-M001"

    @Column(nullable = false, length = 50)
    private String brand;  // 产品品牌，如 "慕思"

    @Column(nullable = false, length = 20)
    private String category;  // 产品品类：智能床垫/电动床/智能枕头

    @Column(nullable = false, length = 20)
    private String protocol;  // 通信方式：MQTT/BLE

    @Column(nullable = false, length = 20)
    private String status;  // 状态：DEVELOPING/PUBLISHED

    @Column(nullable = false, updatable = false)
    private LocalDateTime createTime;

    @Column(nullable = false)
    private LocalDateTime updateTime;
}
```

## 8. 术语表

| 术语 | 说明 |
|-----|------|
| PID | 产品唯一标识符，由系统自动生成，格式：PID_xxxxxx |
| 物模型 | 设备功能的抽象定义，描述设备能做什么 |
| 标识符(identifier) | 每个功能点的唯一标识符，如 "temperature" |
| 功能类型 | 属性(Property)/事件(Event)/命令(Command) |
| 数据类型 | int/float/bool/string/enum |
| 读写类型 | 只读(readOnly)/可写(writeOnly)/可读可写(readWrite) |
| 品类物模型 | 每个产品品类（智能床垫/电动床/智能枕头）的标准物模型 |
| 产品状态 | 开发中(DEVELOPING)/已发布(PUBLISHED) |
| MQTT | 一种物联网通信协议，长连接 |
| BLE | 蓝牙低功耗协议 |
| 驼峰命名 | 首字母小写，后续单词首字母大写，如 "tempTarget" |
| deviceKey | 设备密钥，用于设备认证，格式：DK_PID前6位_8位随机HEX |
| OTA | Over-The-Air，空中升级，远程固件更新 |
| 固件(Firmware) | 设备端运行的软件程序 |
| 升级任务 | OTA升级的执行单元，包含目标固件和目标设备 |
| Sparkline | 迷你趋势图，用于KPI卡片中展示数据趋势 |

## 10. 技术架构

### 10.1 服务拆分

| 服务 | 端口 | 职责 |
|-----|------|------|
| user-service | 8081 | 用户管理（已实现，导航已移除） |
| product-service | 8082 | 产品管理 + 产品分类 |
| thing-model-service | 8083 | 物模型管理 + 模板管理 + 功能点管理 |
| device-service | 8084 | 设备管理 + 上下线 + 属性更新 |
| ota-service | 8085 | 固件管理 + 升级任务管理 |

### 10.2 前端API代理配置

前端通过 Vite 代理转发请求到各后端服务，规则顺序：
```
/api/thing-models → thing-model-service:8083
/api/thing-model  → thing-model-service:8083
/api/ota          → ota-service:8085
/api/devices      → device-service:8084
/api/users        → user-service:8081
/api              → product-service:8082 (兜底)
```

### 10.3 数据持久化

所有服务使用 H2 文件模式持久化：
- product-service: `./data/product-db`
- thing-model-service: `./data/thing-model-db`
- user-service: `./data/user-db`
- device-service: `./data/device-db`
- ota-service: `./data/ota-db`
